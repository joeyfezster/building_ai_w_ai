Key Decisions

▼
1
Auto-detect repo slug from git remote instead of hardcoding
Replace hardcoded 'joeyfezster/building_ai_w_ai' with _get_repo_slug() that parses git remote origin, with --repo CLI override.

These packages are designed to be used by others in their own repos. Hardcoding the monorepo slug breaks all GitHub API calls (PR URLs, GraphQL queries, issue creation) when running from a standalone clone. Auto-detection from git remote is the natural default — it works in both the monorepo and any fork/clone. The --repo flag provides an escape hatch for CI environments or repos with unusual remote configurations. The function is duplicated across three files rather than extracted to a shared module; this is a deliberate choice documented in the plan (shared module is tracked as a separate issue #23) to keep packages dependency-free from each other.

2
Generic path-rewriting rules with verification gate in CI
Publish workflow uses two generic sed rules (strip self-prefix, rewrite sibling cross-refs) plus a grep-based verification gate, instead of maintaining a brittle file-by-file rewrite list.

The monorepo uses paths like 'packages/dark-factory/scripts/...' which are correct in-context but broken in standalone repos where 'scripts/' is at root. Rather than maintaining an ever-growing list of specific sed patterns that must be updated every time a file is added, the design uses two generic rules: Rule 1 strips the package's own prefix (e.g., 'packages/pr-review-pack/' becomes ''), and Rule 2 rewrites cross-references to sibling packages as absolute GitHub URLs. The verification gate (grep for any remaining 'packages/...' references) catches anything the rules miss — making the system open to new files without needing CI changes. The CONTRIBUTING.md exclusion is intentional (it legitimately references the monorepo).

3
Document all runtime dependencies in READMEs and requirements.txt
Added pyyaml to a new requirements.txt, documented gh CLI in both READMEs, and split dark-factory prerequisites into required vs optional.

Users cloning the standalone repos had no way to know pyyaml was needed (scaffold_review_pack_data.py imports it) or that gh CLI must be authenticated (multiple scripts call it). The dark-factory package also uses optional tools (ruff, radon, vulture, bandit) for code quality gates that degrade gracefully — these were undocumented. Making the dependency story explicit is the minimum bar for standalone packages that claim to be installable.

Post-Merge Items

▼
P3 _get_repo_slug() called inside loop in create_postmerge_issues.py

## scripts/create_postmerge_issues.py, L196-201 created = 0 for item in items: repo_slug = _get_repo_slug(args.repo) url = create_issue(item, args.pr, repo_slug, dry_run=args.dry_run) if url: created += 1
Failure scenario
Each loop iteration spawns a subprocess to run 'git remote get-url origin'. With many post-merge items this adds unnecessary latency. Not a correctness bug but a code smell that contradicts the clean patterns in the other two scripts.
Resolution
Hoist repo_slug = _get_repo_slug(args.repo) above the for-loop, matching the pattern used in persist_decisions.py (line 242) and scaffold_review_pack_data.py.

P3 Duplicated _get_repo_slug() across three files

## packages/dark-factory/scripts/persist_decisions.py, L36-48 def _get_repo_slug(override: str | None = None) -> str: """Return owner/repo from CLI flag or git remote origin.""" if override: return override url = subprocess.check_output( ["git", "remote", "get-url", "origin"], text=True ).strip() # SCP-style (git@host:owner/repo.git) has no scheme prefix if ":" in url and not url.startswith(("https://", "http://", "ssh://")): slug = url.split(":")[-1] else: slug = "/".join(url.split("/")[-2:]) return slug.removesuffix(".git")
Failure scenario
A bug fix to URL parsing logic (e.g., handling a new remote URL format) must be applied to all three copies independently. Risk of divergence if one copy is missed.
Resolution
Extract _get_repo_slug() into a shared utility module (tracked as issue #23). All three scripts import from the shared location.

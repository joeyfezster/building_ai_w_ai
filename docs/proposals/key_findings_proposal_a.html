Scope mutation uses undocumented Starlette private API with no stability guarantee
AD core
▶

The implementation injects exception handlers into scope['starlette.exception_handlers'] — a key that is not part of Starlette's public API. If Starlette changes this internal mechanism, the feature silently breaks with no compile-time or test-time warning.

Files: fastapi/routing.py
C
include_router drops self.exception_handlers when flattening already-decorated routes
AD core
▶

When a sub-router's routes are flattened into the parent, the parent router's own exception handlers are not merged into the decorated routes. This means exception handlers set on the parent are silently lost for included routes.

add_exception_handler does not affect routes registered before it is called
CH core tests
▶

Exception handlers added via add_exception_handler() after routes are already registered will not apply to those existing routes, because the scope-injection happens at route decoration time, not at request time. This ordering dependency is undocumented.

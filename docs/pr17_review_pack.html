What Changed

▼

Generated from git diff by delegated diff-reading agent. Code diffs are ground truth.

Product: Switches from DQN to PPO for self-play training. DQN flatlined after 1M+ steps (Q-values differ by ~0.001 across actions, argmax = noise). PPO outputs action probabilities via softmax, sidestepping the value-based limitation. Adds Nature-CNN actor-critic network with orthogonal initialization, GAE-Lambda advantage estimation, clipped surrogate objective, and 10x stronger reward shaping (hit_reward=0.1, rally_bonus_per_step=0.02). Implements Fictitious Self-Play with a 20-checkpoint pool (80% historical / 20% current) for opponent diversity.

Configurable reward shaping with 10x stronger signals

  • Added hit_reward (default 0.1, up from hardcoded 0.01) and rally_bonus_per_step (0.02) to MiniPongConfig dataclass
  • Hit reward now uses self.config.hit_reward instead of hardcoded 0.01 constant
  • Added per-step rally bonus when reward_shaping is enabled -- agent receives +0.02 for every step it survives, incentivizing paddle tracking

Actor-Critic network architecture and on-policy rollout buffer

  • New ActorCriticNetwork: Nature-CNN backbone (3-layer conv) with separate actor (policy logits) and critic (scalar value) heads
  • Orthogonal weight initialization via _layer_init() with gain=sqrt(2) for conv/hidden layers, 0.01 for policy head, 1.0 for value head -- PPO standard practice from Andrychowicz et al.
  • New RolloutBuffer: fixed-size on-policy buffer that collects n_steps of (obs, action, reward, done, value, log_prob), then computes GAE-Lambda advantages
  • GAE computation uses reversed iteration with proper terminal masking (next_non_terminal = 1 - done) for correct bootstrapping at episode boundaries
  • Minibatch generation: random permutation split with per-batch advantage normalization (zero-mean, unit-variance)

PPO agent with clipped surrogate objective and LR annealing

  • New PPOAgent with PPOConfig dataclass (clip_epsilon=0.1, entropy_coef=0.02, vf_coef=0.5, n_epochs=4, n_minibatches=4)
  • act() samples from Categorical distribution over logits -- stochastic exploration, no epsilon-greedy needed
  • act_deterministic() uses argmax for evaluation -- greedy policy without exploration noise
  • update() implements clipped surrogate objective: loss = max(ratio*A, clip(ratio)*A) + vf_coef*vf_loss - entropy_coef*entropy
  • Linear LR annealing from lr to 0 over lr_anneal_total_steps (enabled when lr_anneal=true in config)

PPO self-play training loop with Fictitious Self-Play and dual evaluation

  • New train_ppo_selfplay.py: 365-line training loop collecting n_steps rollouts, computing GAE, running n_epochs of PPO updates
  • PPOSelfPlayOpponent: mirror-flips observations (np.flip axis=1) and maintains its own frame stack -- symmetric view for left/right paddle parity
  • Fictitious Self-Play: checkpoint pool of size 20, opponent sampled 80% from pool / 20% current weights every 50K steps -- prevents co-adaptation to a single opponent version
  • Dual evaluation: evaluate_ppo_policy() runs 10 episodes across 5 seeds vs rule-based opponent; evaluate_ppo_selfplay() runs 100 rallies of self-play to measure rally length
  • evaluate.py gains 111 lines: evaluate_ppo_policy() and evaluate_ppo_selfplay() mirror the DQN evaluation functions but use PPOAgent with deterministic action selection
  • Progress reporting with speed, ETA, loss breakdown (pg_loss, vf_loss, entropy, clipfrac)

Comprehensive smoke tests for PPO components and training loop

  • test_ppo_agent_smoke.py: 122 lines covering ActorCriticNetwork forward shapes, PPOAgent act/act_deterministic/get_value/update, RolloutBuffer add/compute_advantages/get_minibatches/reset
  • test_ppo_training_smoke.py: 67 lines testing PPOSelfPlayOpponent action validity and weight loading, plus a 256-step integration test of the full training loop
  • Tests use real network forward passes with actual tensor shapes -- no mocking of the system under test

PPO self-play training configuration

  • New configs/ppo_selfplay_1m.yaml: 33-line config specifying all PPO hyperparameters (n_steps=128, n_minibatches=4, n_epochs=4, lr=2.5e-4 with annealing), reward shaping (hit_reward=0.1, rally_bonus=0.02), self-play settings (pool_size=20, update_period=50K), and evaluation schedule (every 100K steps)

Key Decisions

▼
1
DQN to PPO algorithm switch
DQN flatlined after 1M+ steps: Q-values converged to near-identical values across all actions (difference ~0.001), making argmax selection equivalent to noise. This is a known failure mode of value-based methods in symmetric environments where the optimal policy is stochastic. PPO's policy gradient approach outputs action probabilities via softmax, naturally producing stochastic policies without epsilon-greedy, and updates on-policy which avoids the replay buffer staleness problem that compounds in self-play settings.

3
Fictitious Self-Play with checkpoint pool
Pure self-play (opponent = current weights) causes co-adaptation: the agent and opponent evolve in lockstep, developing brittle strategies that exploit each other's specific weaknesses but fail against novel opponents. Fictitious Self-Play maintains a pool of 20 historical checkpoints and samples opponents 80% from the pool / 20% current weights, approximating a best-response to the historical strategy mixture. This prevents forgetting earlier skills while still adapting to the latest opponent version.

P1 PPO training oscillation: 700K peak regressed at 800K

Hit ratio peaked at 0.708 (step 700K) but dropped to 0.528 at step 800K before partially recovering to 0.667 at 900K. Self-play rally length shows the same pattern: peak 4.0 at 700K, drop to 1.1 at 800K. This oscillation suggests the opponent pool transition or LR annealing schedule is destabilizing late-stage training. Consider: (1) slower opponent update period in late training, (2) cosine annealing instead of linear, (3) early stopping at peak performance.

## frac = 1.0 - global_step / self.config.lr_anneal_total_steps frac = max(frac, 0.0) lr = self.config.lr * frac
Failure scenario
Training runs past the optimal checkpoint, final model performs worse than the 700K checkpoint. In a 2M+ step run, oscillation could compound into catastrophic forgetting.
Resolution
Identify the optimal checkpoint selection strategy (e.g., save best by eval hit_ratio) and use a more stable LR schedule for longer runs.
P1 Entropy collapse at end of training (clipfrac=0.000)

The PR description mentions clipfrac reaching 0.000 at the end of training. Zero clipfrac means the policy is changing so little that no ratios exceed the clip boundary -- the policy has converged to a near-fixed distribution. Combined with LR annealing to near-zero, the agent cannot recover from a bad opponent pool sample in late training. Consider monitoring entropy throughout training and implementing an entropy floor or adaptive entropy coefficient.

Failure scenario
In longer runs (2M+ steps), entropy collapse makes the agent unable to adapt to new opponent strategies introduced via the checkpoint pool, leading to permanent performance degradation.
Resolution
Add entropy monitoring with a floor (e.g., min_entropy=0.01) that adjusts entropy_coef upward if entropy drops below threshold.
P2 play_minipong.py does not support PPO checkpoints

The interactive play script (play_minipong.py) only supports DQN Q-networks. Loading a PPO checkpoint (which contains an ActorCriticNetwork state_dict under key 'model') will fail with a state_dict mismatch. Need to add PPO agent loading path with --agent-type ppo flag or auto-detection based on checkpoint keys.

Failure scenario
Users cannot visualize or interact with trained PPO agents, making it harder to qualitatively assess training results.
Resolution
play_minipong.py auto-detects checkpoint type (DQN vs PPO) and loads the appropriate agent, enabling interactive evaluation of PPO-trained agents.
P2 Self-play evaluation uses deterministic policy for both sides

evaluate_ppo_selfplay() uses act_deterministic (argmax) for both agent and opponent. Since both sides use the same network and deterministic actions, the outcome for a given seed is entirely predetermined. This makes self-play rally length a measure of the policy's deterministic trajectory length, not its robust play quality. Consider using stochastic policy (act()) for opponent to better measure the agent's robustness.

Failure scenario
Self-play metrics overstate or understate the agent's actual playing strength because they measure a single deterministic trajectory rather than expected performance.
Resolution
Self-play eval uses stochastic opponent policy, providing a more meaningful signal about the agent's average self-play quality.

Key Decisions

▼
1
Scope-injection via Starlette private API rather than middleware wrapping
Injecting handlers into scope['starlette.exception_handlers'] avoids adding middleware layers per router, keeping the ASGI call stack flat and avoiding performance overhead from nested middleware.

The implementation injects router-level exception handlers by writing to scope['starlette.exception_handlers'] inside each route's ASGI closure, rather than wrapping each route in an additional ExceptionMiddleware instance. This is a deliberate trade-off: it avoids the cost of an extra middleware layer per router (which would add ASGI call depth proportional to the number of routers), but it creates a dependency on an undocumented Starlette internal (starlette._exception_handler module and the starlette.exception_handlers scope key). Multiple reviewers (code-health, adversarial, security) independently flagged this coupling. The approach is consistent with how FastAPI already depends on wrap_app_handling_exceptions from the same private module, so the incremental risk is modest — but it doubles the private-API surface area.

2
Handlers baked at registration time rather than resolved at dispatch time
Registration-time baking avoids per-request dictionary lookups and makes each route's behavior deterministic from the moment it is registered, at the cost of making add_exception_handler() order-dependent.

When a route is registered via @router.get() or add_api_route(), the current state of router.exception_handlers is snapshot into the route object. Similarly, include_router() merges the sub-router and route handlers into a new dict that is baked into the flattened route. This means exception handlers are frozen at registration time. The consequence — flagged by both code-health and adversarial reviewers — is that add_exception_handler() called after routes are registered will NOT retroactively apply to those routes. The only test for add_exception_handler registers the handler before the route, so this ordering trap is untested and undocumented.

This is an intentional simplicity trade-off: the feature ships without an override-protection mechanism, keeping the API surface small.

CHCCode Health

The core mechanism by which router-level exception handlers take effect is injecting the merged handler dictionaries into the ASGI scope dict under the key 'starlette.exception_handlers' (routing.py lines 119, 171). This key is read by starlette._exception_handler.wrap_app_handling_exceptions, which is itself imported from a private module (starlette._exception_handler, routing.py line 75).

Neither 'starlette.exception_handlers' as a scope key nor the tuple layout (exc_handlers_dict, status_handlers_dict) are part of Starlette's public API or the ASGI specification. They are internal implementation choices that could be renamed, restructured, or removed in any Starlette release without a deprecation warning.

The feature works correctly against the current Starlette version, but it is essentially relying on two private APIs simultaneously: the module starlette._exception_handler (already acknowledged with an existing import) and the scope key convention. If Starlette ever changes the tuple layout to a namedtuple, a dict, or a different key name, the exception handler injection will silently do nothing — no runtime error, no warning, just handlers that stop firing.

A more resilient approach would be to wrap route apps in a lightweight ASGI middleware layer that installs handlers via a supported Starlette mechanism, rather than reaching into scope internals. Alternatively, the dependency on the scope key format should be documented explicitly with the Starlette version it targets, so future Starlette upgrades trigger a deliberate re-check.

APIRouter.add_exception_handler (routing.py lines 1364–1369) simply does self.exception_handlers[exc_class_or_status_code] = handler. However, when a route is registered via @router.get or router.add_api_route, the exception_handlers snapshot is baked into the APIRoute instance at that moment (routing.py line 874: self.exception_handlers = exception_handlers or {}) and into the route's ASGI app (line 1007). Routes registered before the add_exception_handler call are not retroactively updated.

The only test for add_exception_handler (test_add_exception_handler_method, test file lines 190–204) registers the handler before the route, so the hidden ordering requirement is never exercised. A developer who writes:

router = APIRouter()

@router.get('/fail')
def fail(): raise MyException()

router.add_exception_handler(MyException, handler)  # too late

will see the handler silently do nothing for that route. The docs describe add_exception_handler as working "the same as passing exception_handlers in the constructor" — which is only true if routes are registered after the call.

The fix is either: (a) make add_exception_handler raise an error if routes are already registered, (b) add a test and doc note about the ordering requirement, or (c) redesign to apply handlers at dispatch time rather than registration time.

Both request_response (lines 108–119) and websocket_session (lines 160–171) contain an identical block that reads scope.get('starlette.exception_handlers', ({}, {})), builds merged_exc and merged_status dicts by iterating exception_handlers.items() and branching on isinstance(key, int), then writes the result back into scope. The block is character-for-character identical.

This is a classic duplication-by-copy-paste. A helper function — something like _inject_exception_handlers(scope, exception_handlers) — would eliminate the redundancy and make it easier to evolve the merge semantics (e.g., if Starlette's internal tuple layout ever changes).

The duplication is low-risk today but creates a maintenance trap: a reviewer fixing a bug in one copy must remember to fix the other.

APIRouter.add_api_route (routing.py line 1421) accepts an exception_handlers parameter that allows per-route handler overrides on top of the router's handlers. However, the api_route() decorator (lines 1481–1541) and all HTTP method shortcuts (get, post, put, delete, patch, options, head, trace) do not expose this parameter — they call add_api_route without it.

This means a user who wants to override the router-level handler for a single route cannot do so via decorator syntax:

# This is the only way to set a per-route override — awkward and undiscoverable
router.add_api_route('/special', endpoint, exception_handlers={MyExc: special_handler})

# This does not work — no exception_handlers parameter available
@router.get('/special', exception_handlers={MyExc: special_handler})  # TypeError
def endpoint(): ...

The omission is an incomplete interface: the capability exists in the plumbing but is not surfaced at the level users interact with. The docs only show the router-level parameter, so users have no hint that route-level overrides are even possible. If the feature is intentionally not exposed at the decorator level, that decision should be documented; if it's an oversight, the parameters should be added.

SEB+Security

The merge logic in APIRouter.include_router() (lines 1832-1838 and 1890-1894) applies route.exception_handlers on top of router.exception_handlers via dict.update(). This gives innermost-defined handlers unconditional precedence over outer handlers for the same exception type.

Security-adjacent concern: if an outer router defines a handler for a security-sensitive exception (e.g., a custom authentication error) and an inner router accidentally or maliciously redefines the same exception type, the outer handler is silently replaced. In the FastAPI open-source context (external contributors), this means a contributed router could suppress a security exception handler defined in the parent application.

This is an architectural design choice, not a direct vulnerability — it mirrors how CSS specificity and middleware ordering work. The risk is B+: limited exploitability (requires control over router composition), but no safeguard or documentation warning exists for this behavior pattern. A note in the docs about the override implications for security-relevant handlers would improve the security posture.

The core security question for the scope mutation pattern is whether scope["starlette.exception_handlers"] modifications in one request can leak into another. ASGI scopes are per-connection dicts created by the server for each request; they are not shared across requests. The PR correctly creates new merged dicts (merged_exc = {**existing_exc}, merged_status = {**existing_status}) rather than mutating the existing tuples in place, so parent-scope state is not modified. There is no cross-request or cross-user contamination risk.

This pattern mirrors how Starlette itself propagates exception handlers down the middleware stack via scope manipulation, making it consistent with the framework's own security assumptions.

ADCAdversarial

In both request_response() (line 108–119) and websocket_session() (line 160–171), the implementation writes to scope["starlette.exception_handlers"] before delegating to wrap_app_handling_exceptions(). This is the internal handshake by which Starlette's exception handler middleware propagates registered handlers into the request scope.

The problem: starlette.exception_handlers is a private implementation detail of starlette._exception_handler. It is not documented, not typed in Starlette's public API, and not stable across versions. The key name starts with starlette. — the same naming convention Starlette uses for its own private scope entries (e.g., starlette.routing.path_params).

FastAPI already imports wrap_app_handling_exceptions from starlette._exception_handler (a private module, note the underscore). This PR doubles down on that coupling by also depending on the exact tuple structure (exc_handlers_dict, status_handlers_dict). If Starlette ever changes this internal protocol — which it has done before during the ASGI middleware refactor — this feature breaks silently. There are no version guards or defensive checks.

A more stable approach would be to wrap the ASGI app in a lightweight exception handler middleware instance, similar to how Starlette's own ExceptionMiddleware is constructed, rather than injecting into a private scope slot.

When a route is added to router_b via @router_b.get(), add_api_route is called. At that point, it computes current_exception_handlers = dict(router_b.exception_handlers) and stores it in route.exception_handlers. The route object carries the sub-router's handlers baked in.

Later, when router_a.include_router(router_b) is called (lines 1832–1838), the merge logic does:

current = {}
current.update(router.exception_handlers)  # router_b.exception_handlers
current.update(route.exception_handlers)   # already contains router_b.exception_handlers

Both updates write the same handlers. This is redundant in normal cases, but it creates a divergence hazard: if add_exception_handler is called on router_b after routes are added (but before include_router), route.exception_handlers would be stale (missing the new handler) while router.exception_handlers would have it. The merge would then correctly pick up the new handler via the router.exception_handlers path — but this is relying on compensating errors rather than a coherent design. The two sources of truth (route.exception_handlers and router.exception_handlers) can drift, and the merging logic silently papers over that drift.

There is no test for the ordering: add handler to router → add route to router → include router. All tests add handlers to the router constructor, so route.exception_handlers and router.exception_handlers are always in sync.

Consider three routers: router_c (innermost) included in router_b included in router_a included in app.

When router_b.include_router(router_c) is called, routes from router_c are flattened into router_b. At this point, add_api_route is called on router_b with exception_handlers = {router_c_handler, route_handler}. add_api_route then prepends router_b.exception_handlers as base and updates with the passed handlers. Result: route in router_b.routes has exception_handlers = {router_b_handler updated_with router_c_handler} — i.e., router_c_handler wins.

When router_a.include_router(router_b) is called, the now-flattened route is processed again. Its route.exception_handlers already has router_c_handler winning. router_b.exception_handlers is also merged (but is overridden by route.exception_handlers). Then add_api_route on router_a prepends router_a.exception_handlers as base. Final: router_c wins.

This is correct, but it is fragile: the correctness depends on the bottom-up include order being strictly respected. If router_a.include_router(router_b) is called before router_b.include_router(router_c), routes from router_c are NOT yet in router_b.routes, so they are never flattened into router_a. This is a pre-existing FastAPI ordering constraint, but the new exception_handlers feature adds another dimension where the ordering produces silently different results. There is no test or documentation covering this ordering sensitivity.

Handler registration uses Callable[..., Any] throughout — the ... parameter specification means any argument list is accepted and the return type is Any. No runtime validation checks that the handler matches the expected signature (Request, Exception) -> Response.

MEDIUM <code>add_exception_handler()</code> silently fails for already-registered routes

Handlers are baked into routes at registration time. Calling router.add_exception_handler() after routes are decorated has no effect on those routes. The method name implies dynamic behavior but delivers static behavior. No test exercises this ordering, and the documentation does not warn about it. Both code-health and adversarial reviewers flagged this independently.

## fastapi/routing.py, lines 1364-1369 def add_exception_handler(\n self,\n exc_class_or_status_code: int | type[Exception],\n handler: Callable[..., Any],\n) -> None:\n self.exception_handlers[exc_class_or_status_code] = handler

Either: (a) the docs explicitly warn about the ordering requirement, (b) add_exception_handler raises if routes already exist, or (c) handlers are resolved at dispatch time instead of registration time.
